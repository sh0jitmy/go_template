# データベース設計 & 開発ベストプラクティス

## 1. SQLファースト方針 (No-ORM)
- **生のSQLの推奨**: パフォーマンスの不確実性と過剰な抽象化を避けるため、複雑なORMは採用せず、生SQLまたはそれに近いクエリ記述ツールを使用し、開発者が制御可能な明示的な SQL を書く。
- **型安全な SQL クエリ**: スキャン漏れやパラメータのマッピングミスを防ぐため、名前付きプレースホルダーを利用し、動的SQLの生成は最小限にする。

## 2. コネクションプールのチューニング
データベースへの過剰な接続やリソースリークを防ぐため、必ず最大同時接続数、アイドル時接続数、および最大接続生存期間を環境に合わせて最適に設計・設定する。

- **接続数上限の算出**:
  - `MaxOpenConns`（最大同時接続数）は、データベースサーバーが許容できる最大接続数（`max_connections`）を、アプリケーションのインスタンス（レプリカ）数やバッチ処理スレッド数で割った値よりも小さく設定する。
- **アイドル接続数の最適化**:
  - `MaxIdleConns`（アイドル時の最大接続数）は、極力 `MaxOpenConns` と同値に設定することを推奨する。これにより、接続の切断・再作成のスパイクを最小化し、TCPハンドシェイクのオーバーヘッドを削減する。
- **最大生存期間（Lifetime）**:
  - `ConnMaxLifetime`（接続の最大生存期間）は数分〜数十分程度に設定し、長時間の接続によるメモリリークやロードバランサによる強制切断からリソースを保護する。

## 3. トランザクションと分離レベルの管理
- **Context の伝搬**: トランザクション処理には必ずコンテキスト（タイムアウト・キャンセル情報）を伝搬させる。
- **確実なロールバック処理**: トランザクション開始直後にロールバックを遅延実行（defer）し、コミット成功時のみフラグを反転させることで、例外発生時のロールバック漏れを防ぐ。
  ```go
  tx, err := db.BeginTx(ctx, &sql.TxOptions{Isolation: sql.LevelReadCommitted})
  if err != nil {
      return err
  }
  defer tx.Rollback() // コミットされていない場合は自動でロールバックを実行（エラーは安全に無視される）

  // 複数のクエリ処理を実行
  if _, err := tx.ExecContext(ctx, query1, arg1); err != nil {
      return fmt.Errorf("query1 failed: %w", err)
  }
  if _, err := tx.ExecContext(ctx, query2, arg2); err != nil {
      return fmt.Errorf("query2 failed: %w", err)
  }

  // 正常終了時のみコミット
  if err := tx.Commit(); err != nil {
      return fmt.Errorf("commit failed: %w", err)
  }
  ```
- **分離レベルの選択**:
  - デフォルトは `Read Committed`。
  - ファントムリード等の防止や厳密な整合性が必要な決済処理などでは、`Serializable` などの厳格な分離レベルを指定し、必要に応じてアプリケーション側でデッドロック時のリトライロジックを実装する。

## 4. データベースマイグレーション
- **スキーマ管理のコード化**: スキーマ変更は手動で行わず、必ずバージョン管理マイグレーションツールを利用し、SQLファイル（Up/Down）としてリポジトリで管理する。
- **Atlas CLIによるバージョン管理マイグレーション (Versioned Migration)**:
  - マイグレーションツールとして `Atlas CLI` (atlasgo.io) を標準採用する。
  - `ent` スキーマ（または生SQL）から現在のスキーマ状態と変更後のスキーマ状態を Atlas で比較し、自動的にマイグレーションSQLファイル（`migrations/` ディレクトリ）を生成・バージョン管理する。
  - DDLの変更時は、適用前に `atlas migrate lint` などの静的解析を実行し、データ消失（破壊的変更）やパフォーマンス低下を伴うロック等が発生しないかを検証する。
- **マイグレーションの自動実行**: マイグレーションはデプロイパイプラインの初期フェーズ（CI/CDやAnsible適用時）で自動実行されるようにする。
- **後方互換性の維持**: インフラの稼働率を下げることなくデプロイを行うため、マイグレーションは常に「後方互換性（テーブル追加やカラム追加、あるいはカラム削除時の新旧アプリ並行稼働対応）」を考慮して設計する。

## 5. クエリ最適化と監視
- **EXPLAIN による実行計画評価**: インデックスが効いていないフルスキャンを避けるため、クエリチューニング時はデータベース上で `EXPLAIN` や `EXPLAIN ANALYZE` を実行し実行計画を評価する。
- **スロークエリ監視**: 定義した閾値（例: 200ms）を超える SQL クエリは、警告レベルでログ出力し、実行時間と引数を記録し監視する。
- **メトリクス監視**: プロダクション環境では、現在の接続プール状態（オープン接続数、使用中接続数、待機時間）をメトリクスとして公開し、リソースの飽和度 (USEメソッド) を監視する。

## 6. Ent ORM (entgo.io) の設計・運用指針
プロジェクトで `ent` ORM を採用する場合、DBAおよび開発者は以下の設計指針を遵守しなければなりません。
- **接続プールの最適設定**:
  - `ent.Open` 時に返される `*ent.Client` にて、内部ドライバー（`sql.DB`）に対して `database-design:2` で規定された最大同時接続数(25)、アイドル接続数(25)、および最大接続生存期間(5分)を必ず適用すること。
- **型安全スキーマの定義とDB制約**:
  - スキーマ定義において、適切な一意性制約（`Unique()`）やインデックス（`Indexes()`）を明示的に定義し、Goの型安全性とDB側の整合性を両立させること。
- **トランザクション管理と二重登録防止**:
  - 書き込み操作（インサート・更新）では、必ず `client.Tx(ctx)` を用いてトランザクションを明示的に制御し、`defer rollback` パターンを実装すること。
  - 同期再送時の二重インサートを防止するため、`Upsert` (ON CONFLICT DO UPDATE/NOTHING) をスキーマのユニークキー（デバイスIDとタイムスタンプの複合キーなど）に対して正しく構成すること。
- **自動マイグレーションの運用監視**:
  - ローカル開発環境では `client.Schema.Create(ctx)` による自動マイグレーション（Auto-Migration）を許容するが、本番環境のデプロイにおいては、`client.Schema.Create` は使用せず、`ent` のスキーマ定義を Atlas に食わせることで生成されたマイグレーション SQL ファイルを使用し、Atlas CLI (またはそれに準ずるマイグレーション適用プログラム) 経由で適用すること。
  - マイグレーションファイルは CI や本番デプロイ時に事前に DDL 解析とレビュー（手動承認）を経て適用されること。
